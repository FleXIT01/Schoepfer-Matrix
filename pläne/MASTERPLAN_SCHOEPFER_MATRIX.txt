0. KURZFASSUNG (TL;DR)
--------------------------------------------------------------------------------

Deine Intuition ist korrekt: OpenClaw ist das richtige zentrale Hirn.

Grund: OpenClaw ist KEIN kleines Tool, sondern eine vollständige, produktionsreife
Agent-Plattform mit ~130 Extensions und ~70 Skills. Es bringt bereits mit:
  - JEDEN LLM-Provider (anthropic, openai, ollama, deepseek, qwen, mistral,
    google, groq, together, openrouter, lmstudio, vllm, litellm, ...)
  - JEDEN Messenger-Kanal (Telegram, Discord, Slack, Signal,
    iMessage, WeChat, Matrix, Teams, ...)
  - Browser-Steuerung (extensions/browser)
  - Bild/Video/Voice (comfy, fal, runway, pixverse, elevenlabs, deepgram)
  - Web-Suche (duckduckgo, brave, exa, tavily, perplexity, firecrawl, searxng)
  - Coding-Agenten (codex, opencode, kilocode + skill "coding-agent")
  - Memory (active-memory, memory-lancedb, memory-wiki)
  - MCP-Host (skill "mcporter", plugin-sdk) + Canvas (Live-UI)
  - Skill-Selbstbau (skill "skill-creator") + Marktplatz (skill "clawhub")
  - Agent-Harness + Sub-Agenten (packages: agent-core, acp-core, codex-supervisor)

=> Wir bauen NICHTS davon nach. Wir machen OpenClaw zum Gehirn und hängen nur
   das an, was OpenClaw NICHT hat. Das einzig wirklich Einzigartige in deinen
   lokalen Repos ist:
     (A) die WISSENSCHAFTS-APIs (AlphaFold, ChEMBL, PubMed, Ensembl, STRING,
         Reactome, openFDA, RCSB-PDB, arXiv, OpenAlex, EuropePMC)
     (B) die BOT-FABRIK aus bot1 (Architect->Coder->Gates->Fixer mit echter
         Verifikation in einer Sandbox)
     (C) ein paar Spezialdienste (gpt-researcher, repo-critic-ai, deepwiki,
         PDFMathTranslate, local-llm-planner)

Die 4 Verbindungs-Mechanismen (genau diese, nichts anderes):
  1. NATIVE EXTENSION   -> schon in OpenClaw vorhanden, nur konfigurieren
  2. MCP-SERVER         -> eigener Tool-Server (Python), den OpenClaw via
                           "mcporter" / lokalem MCP-Modus konsumiert  [HAUPTWEG]
  3. OPENCLAW-SKILL     -> SKILL.md-Workflow, der MCP-Tools/Services orchestriert
  4. HTTP-SERVICE       -> schwerer lokaler Dienst, von Skill/MCP per HTTP gerufen

Wie "das Richtige je Aufgabe gewählt wird":
  OpenClaws Agent-Loop liest die `description` jedes Skills und das Schema jedes
  MCP-Tools und WÄHLT SELBST anhand dieser Beschreibungen. D.h. das Routing
  entsteht NICHT durch eine if/else-Kette (wie aktuell in bot1/start.py), sondern
  durch präzise Skill-Beschreibungen + Tool-Schemas + einen "Supervisor"-Skill
  für die Mehr-Agenten-Zerlegung.

4. WIE "DAS RICHTIGE" GEWÄHLT WIRD  (das Routing-Modell, genau)
--------------------------------------------------------------------------------

Es gibt KEINE zentrale if/else-Routingdatei mehr. Stattdessen 3 Ebenen:

A) WERKZEUG-EBENE (innerhalb einer Aufgabe):
   OpenClaws agent-core bekommt alle MCP-Tool-Schemas + Skills sichtbar gemacht.
   Die `description` + `input_schema` SIND der Router. Deshalb gilt:
   REGEL: Jedes MCP-Tool und jeder Skill braucht eine präzise, trigger-orientierte
   Beschreibung (Substantiv-Phrase, klare Stichworte). Beispiel:
     science-mcp.alphafold_fetch:
       "3D-Struktur eines Proteins aus AlphaFold per UniProt-ID. Für: Proteinform,
        Domänen, Strukturbiologie."
   So wählt das Modell alphafold_fetch automatisch, sobald es um Proteinstruktur geht.

B) AUFGABEN-EBENE (welcher Skill/Workflow):
   Skills sind grobgranulare Workflows. Ihre `description` triggert sie. Beispiel:
     drug-discovery: "Wirkstoffe/Targets gegen eine Krankheit finden: Literatur,
       Moleküle, Targets, Struktur, Bericht."
   Das Modell erkennt "finde Inhibitoren gegen X" -> Skill drug-discovery.

C) ZERLEGUNGS-EBENE (mehrere Agenten parallel):
   Ein Meta-Skill "supervisor" (Muster aus swarm/agent-framework) zerlegt ein
   großes Ziel in Teilaufgaben, ordnet jeder Teilaufgabe einen Skill/MCP-Tool zu,
   führt unabhängige Teile PARALLEL aus (codex-supervisor / acp-core) und fügt
   die Ergebnisse zusammen. Entscheidungslogik im supervisor:
     - Zähle benötigte Fähigkeiten (Recherche? Code? Bild? Deploy?).
     - 1 Fähigkeit  -> direkter Skill/Tool-Aufruf (kein Overhead).
     - >1 Fähigkeit -> Teilaufgaben-Graph; unabhängige Knoten parallel.
     - Code-Aufgaben immer mit anschließendem review-mcp-Gate.

D) MODELL-WAHL je Teilaufgabe (model-catalog-core):
   - Routing/Klassifikation/kurze Schritte -> kleines lokales Modell (ollama,
     z.B. qwen2.5:14b) = billig/schnell.
   - Synthese/Code/Reasoning -> stärkstes verfügbares Modell (anthropic/openai
     wenn Key vorhanden, sonst größtes lokales).
   - local-llm-planner (planner-mcp) sagt, was in den VRAM passt.

WICHTIGE LEHRE aus bot1: kleine Modelle VERWEIGERN oder HALLUZINIEREN. Deshalb
gilt systemweit das "Completion-Gate"-Prinzip auch in OpenClaw-Skills:
behaupteten Erfolg gegen die Realität prüfen (Datei existiert? Build grün?
Review bestanden?) bevor "fertig" gemeldet wird.

   Tools: build_bot(spec)->Paket+BUILD_REPORT, plan_tools(spec),
          verify_package(path).
   Wert:  echte Sandbox-Verifikation (Import/Smoke/Tool-Gates) — das hat
          OpenClaws coding-agent so nicht.
   Anbindung: MCP. Der Skill "build-bot" ruft diese Tools.

5.3  research-mcp   (gpt-researcher)
   Tools: deep_research(query, depth)->Bericht mit Quellen.
   Anbindung: MCP-Wrapper um gpt-researcher (läuft als HTTP-Dienst dahinter).

5.4  review-mcp   (repo-critic-ai)
   Tools: scan_repo(path)->Findings(severity, file, line, msg).
   Einsatz: Pflicht-Gate nach jeder Code-Generierung.

5.5  planner-mcp   (local-llm-planner)
   Tools: resources(), can_load(model), recommend(task).
   Einsatz: OpenClaw fragt vor schweren Schritten (ComfyUI/großes Modell).

5.6  pdf-mcp   (PDFMathTranslate)   — optional Phase 2
   Tools: translate_pdf(path, target_lang)->übersetztes PDF.

5.7  Skills (SKILL.md-Workflows in OpenClaws skills/ bzw. Workspace):
   - supervisor ....... Zerlegung + Mehr-Agenten-Orchestrierung (Kern!)
   - drug-discovery ... Krankheit/Target -> Literatur+Moleküle+Struktur+Bericht
                        (ruft science-mcp + research-mcp)
   - deep-research .... Thema -> umfassender Bericht (research-mcp + Suche + memory)
   - build-bot ........ Spezifikation -> verifiziertes Bot-Paket (factory-mcp
                        + coding-agent + review-mcp)
   - repo-review ...... Repo -> Review-Bericht (review-mcp + browser-Test)
   - repo-to-wiki ..... Repo -> Architektur-Wiki (deepwiki)
   - paper-translate .. PDF -> Übersetzung (pdf-mcp)
   - knowledge-ask .... Frage -> Antwort aus dem Lern-Korpus (MaxKB/WeKnora)
   Jeder Skill: knappe `description` (Trigger), Body = Schritte, `requires` =
   benötigte Bins/MCP/Config. skill-creator hilft beim Anlegen.

5.8  Wissens-Ingestion (einmalig + inkrementell):
   - Skript indexiert die [KORPUS]-Repos in MaxKB/WeKnora.
   - "Lerneffekt": Ergebnisse abgeschlossener Aufgaben (Berichte, Reviews,
     Builds) zurück in die Wissensbasis schreiben (wie der heutige
     _ingest_cycle_learnings, aber nach OpenClaw-memory + RAG).

6. ROUTING-KATALOG  (Aufgabe -> Pipeline; durchgerechnete Beispiele)
--------------------------------------------------------------------------------

"Finde Inhibitoren für EGFR bei Lungenkrebs"
  -> Skill drug-discovery
  -> science-mcp.chembl_search(target) + .pubmed_search + .europepmc_search
     + .alphafold_fetch(EGFR=P00533) + .string_db(EGFR) + .reactome_search
  -> Synthese (starkes Modell) -> Bericht -> memory + Antwortkanal.

"Suche Infos über X und speichere als txt"
  -> direkter Tool-Pfad (1 Fähigkeit): native Suche (duckduckgo/exa)
     -> Datei schreiben -> Completion-Gate (Datei existiert?) -> fertig.

"Baue eine Android-App zur Protein-Faltungserkennung"
  -> Skill supervisor (mehrere Fähigkeiten)
  -> [recherche: science-mcp+research-mcp] [mockup: comfy] [code: coding-agent
     /factory-mcp] [review: review-mcp + browser] [deploy: build-bot/shell]
     [doku: repo-to-wiki] -> APK + Bericht zurück.

"Übersetze dieses Paper (PDF) und fasse es zusammen"
  -> Skill paper-translate (pdf-mcp) -> Skill summarize (nativ) -> Antwort.

"Reviewe mein Repo und bau eine Doku-Webseite"
  -> supervisor: review-mcp (parallel) + repo-to-wiki (parallel) -> Zusammenführung.

"Wie entwerfe ich ein skalierbares Chat-System?"
  -> Skill knowledge-ask -> RAG über system-design-primer (Korpus) -> Antwort.

  - Verifizieren: OpenClaw antwortet über einen Channel.
  - Ergebnis: das echte Hirn lebt (statt Python-Fake).

  - MCP-Server um die 11 fertigen Science-Tools bauen.
  - via mcporter in OpenClaw registrieren.

PHASE 2 — Kern-Skills + factory-mcp + review-mcp
  - Skills drug-discovery, deep-research, build-bot, repo-review anlegen.
  - factory-mcp (bot1-Fabrik) + review-mcp (repo-critic-ai) als MCP.
  - Test: kompletter Bot-Build mit Verifikation über einen Channel.

PHASE 3 — supervisor (Mehr-Agenten) + Modellwahl
  - supervisor-Skill (Zerlegung, Parallelität, Gates).
  - model-catalog: kleines Modell fürs Routing, großes für Synthese/Code.
  - planner-mcp für lokale VRAM-Entscheidungen.

PHASE 4 — Wissen + Medien + Deployment
  - Korpus-Repos in MaxKB/WeKnora indexieren; knowledge-ask-Skill.
  - comfy (ComfyUI) + Voice/Video-Extensions aktivieren.
  - Deploy-Wege (firebase/android/docker) als Skill-Schritte.

PHASE 5 — Selbstverbesserung + Marktplatz
  - skill-creator nutzen: das System baut neue Skills für wiederkehrende Abläufe.
  - clawhub: fertige Skills registrieren/teilen.
  - repo-to-wiki + deepwiki für automatische Doku.

  - KEINE Behauptung, dass schwere Dienste (MaxKB, ComfyUI, SuperAGI,
    agenticSeek) "von allein" laufen — sie müssen je einmal eingerichtet werden;
    bis dahin degradieren die zugehörigen Skills sichtbar (kein Fake-Erfolg).
